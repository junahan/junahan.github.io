Fabric 2.0 新特性介绍

Fabric 第一个重要发布版本是 Fabric 1.0。Fabric 2.0 则是最新发布版本，该版本 为用户和管理员交付了一些重要的新特性和变化。这些新特性包括支持新的应用和隐私方案、增强智能合约治理、营运节点的新选项等。保持不变的是根据你自己的需要升级网络组件的能力、支持从 v1.4.x 版本滚动升级以及只有组织成员准备好的情况下激活新功能的能力。

让我们来快速浏览下 Fabric 2.0 发布版的亮点吧！

  • 新链码生命周期管理: 完全重新设计链码生命周期管理流程，提供安装链码至对等节点和在通道上激活链码的新流程，从而引入智能合约的去中心化治理。新链码生命周期管理允许多个组织一道对链码的参数达成一致，如，链码背书策略。
  • 为协作和共识而生的新链码应用模式: 用于增强链码生命周期管理的去中心化方法同样适用于链码应用程序以确保组织能够在交易数据被提交到账本之前对其达成一致。额外的，组织现在能够自行扩展链码副本以满足他们自己的需求。例如，一个组织能够添加他自己的链码校验逻辑以更好的保护其特定的利益。对于既定的一笔交易，只要必要的背书节点之间就链码的执行结果达成共识，交易就想以前的版本一样被提交至账本。
  • 基于必要原则的数据隐私保护: 除了以往使用通道和私有数据集合在网络成员间共享数据的方法外，v2.0 提供新的非指定组织私有数据集合特性以允许组织间基于必要原则共享私有数据。这个特性可以消除为共享私有数据而在组织成员间创建大量的通道或者私有数据集合组合。通道账本上的 Hashes 作为交易证据以确保当私有数据在共享的时候能够被验证或者需要的时候被审计。
  • 新的外置链码启动器: 外置链码启动器允许管理员使用选择的技术构建和启动链码。去掉了对等节点访问容器守护进程的要求。例如，管理员可以将链码作为一个外部服务部署于 Kubernetes pod 上。
  • 新的 Raft 共识类型: 在 v1.4.1 版本引入的排序服务 Raft 共识现在是推荐使用共识服务。其消除了对外部 Kafka 集群的依赖。这个特性支持由多个组织提供排序服务的去中心化治理模式。额外的，排序节点现在能够选择为哪通道提供排序服务代替传统的必须为所有通道提供服务的模式。这带来了一个重要的可扩展性提升以更好的支持大量通道和交易，更不用说很明显的隐私保护的提升。
  • 性能提升: 加入几个贯穿 v1.x 和 v2.0 的性能提升。加入新的缓存以提升使用 CouchDB 状态数据库的对等节点的性能。另外，交易验证也在提交阶段并发执行。优化排序节点消息处理流以去掉冗余检查、并且写锁处理现在是异步的。链码查询支持分页以提升大结果集性能。现在是时候说系统可以达到数千 TPS 吞吐量。

1 智能合约去中心化治理

Fabric v2.0 为智能合约引入了去中心化治理，提供安装链码至对等节点和在通道上激活链码新处理流程。新链码生命周期管理允许多个组织一道对链码的参数达成一致，如，链码背书策略。相对于此前的链码生命周期管理，新模型提供了一些提升。

  • 多组织必须就链码的参数达成一致: 在 Fabric v1.0 版本中，一个组织有能力为通道中所有组织设置链码参数（通过实例化背书策略），其他组织成员仅仅有拒绝安装链码并不参与交易的权利。而新的链码生命周期管理更加灵活，既支持中心化模式（和老版本的一样），也支持去中心化模式，新模式要求链码在通道中激活之前，有足够多的组织就链码的背书策略和其他细节达成一致。
  • 更加考虑周全的链码升级流程: 之前版本的链码生命周期管理允许单一组织发起链码升级交易，这会给其他仍没有安装新链码的组织带来风险。新模型中，只有足够多的组织同意升级链码的情况下才允许升级链码。
  • 简单背书策略和私有数据集更新: 新的链码生命周期模型允许修改背书策略或者私有数据集配置配置而无需重新打包和安装链码。用户也可以从一个新的要求通道中大多数组织背书的默认策略中获益。新策略会随着通道中组织的加入和退出而自动更新。
  • 方便检出的链码包: 链码安装包以容易读取的 TAR 文件打包。这使得安装包很容易检出及利于跨不同组织协调一致安装链码。
  • 使用一个安装包在通道中启动多个链码: 在之前版本中，当链码包被安装的时候，通过指定名字和版本号来标示通道中的每个链码。现在则可以使用单一的链码包在相同或者多个通道中使用不同的名字部署多次。例如，可以使用同一份链码的不同副本来追踪不通类型的资产。
  • 不同成员的链码包不必保持一摸一样: 组织可以根据自己的使用场景扩展链码，例如，组织可以实施不同的验证规则。链码交易只要有必要数量的组织背书通过就可以被验证并提交到账本中。这也允许单个组织根据自己的计划执行下版本修复而不要求锁定整个网络。

1.1 使用新的链码生命周期

对于现有的 Fabric 部署，Fabric 2.0 仍然可以继续沿用以前的链码生命周期。只有通道应用能力 (Channel Application Capability) 升级到 v2.0 版本的时候，新生命周期才会有效。可以查看 链码管理员文档 (Chaincode for Operators)了解有关新链码生命周期的详细细节。

2 为协同和共识而生的新链码应用模式

用于增强链码生命周期管理的去中心化方法同样适用于链码应用程序以确保组织能够在交易数据被提交到账本之前对其达成一致。

  • 自动化检查: 如前所述，组织可以扩展链码以在为交易背书之前增加自动检查功能。
  • 去中心化协议 (Decentralized agreement): 人类的判断可以被建模到跨越多个交易的链码处理流程中。链码可能要求一些组织行动人表明对一个账本交易的条款达成一致。然后，一个最终的链码提议能够验证所有单个交易都符合交易条款并最终在通道成员间执行结算。有关私有条款说明的现实例子，请参阅 Private data 文档里的资产交易场景示例。

3 私有数据增强

Fabric 2.0 支持私有数据处理及共享的新模式而不必为通道內所有需共享的成员创建私有数据集组合。特别是你可能想跨越不同的数据集合而不是一个多成员集合间共享数据，每个集合都可能包含一个单一的组织，或者包含一个单一组织及监管或者审计机构。

Fabric 2.0 提供好几个增强以使得新的私有数据新模式成为可能。

  • 共享和验证私有数据: 当分享私有数据给不在数据集內的通道成员或者给另外一个包含一个或者多个成员的私有数据集（通过向私有数据集写入 KEY 的方式）的时候，接收方可以利用 GetPrivateDataHash() 链码 API 来验证私有数据和链上哈希匹配且由前序交易的私有数据所创建。
  • 集合级别的背书策略: 作为一个选项，私有数据集现在可以背书策略定义，该定义覆盖链码级别的背书策略。这个特性能够用来约束那些组织可以写数据到私有数据集中，并激活此前提到的新的链码生命周期和链码和应用模式。例如，可以有一个链码级别的背书策略，其要求大多数组织进行背书，但是对于既定的交易，也许需要参与交易的两个组织分别在他们自己的私有数据集上达成一致并进行背书。
  • 隐含的按组织 （per-organization） 内置私有数据集合: 在 Fabric v2.0 中部署链码的时候，如果想使利用按组织内置私有数据模式，甚至都无需定义私有数据集合。隐含的组织特定集合无需定义即可使用。

要了解更多有关私有数据模式，请参阅 Private data (概念文档)。有关更多私有数据集合配置和隐含数据集合的信息，请参阅 Private data (参考文档)。

4 外部链码启动器

外部链码启动器允许管理员使用他们选择的技术构建和启动链码。使用外部链码构建器和启动器不需要像此前版本一样使用 Docker 容器 API 启动和运行链码。

  • 移除对 Docker 守护程序的依赖: Fabric v2.0 前序版本要求对等节点能够访问 Docker 守护进程以构建和启动链码 - 在一些情形下，生产环境可能不满足 Peer 进程所要求的权限。
  • 替代容器选择: 链码已经不再要求运行于容器之中，其可能在管理员选择的环境中被执行（包含容器）。
  • 外置构建器可执行程序: 管理员可以外置一组构建器可执行程序来代替 Peer 节点构建和运行链码。
  • 链码可作为外部服务: 传统方式中，链码容器由 Peer 节点启动并随后连接回 Peer 节点。现在可以将链码作为服务来运行，例如，链码可以运行于 Kubernetes Pod 中，Peer 节点可以连接并执行链码。更多信息，请参阅链码作为服务。

参阅外部构建器和启动器一节以了解更多有关外部链码启动器特性。

5 提升 CouchDB 性能的状态数据库缓存

  • 但使用外部 CouchDB 状态库的情形下，由于背书和验证阶段导致的读延迟性能瓶颈已成成为历史。
  • 在 Fabric v2.0 中，一个新的 peer 缓存使用快速本地缓存读取模式代替这些昂贵的查找操作。缓存大小可以通过 core.yaml 属性 cacheSize 进行配置。

6 基于 Alpine Linux 的容器镜像

自 Fabric v2.0 开始，Fabric Docker 镜像将采用 Alpine Linux - 一个面向安全的轻量级 Linux 发行版。这意味着 Docker 镜像的尺寸更小巧、可更快速的下载、减少启动时间、也减少占用主机磁盘空间。Alpine Linux 被设计为自第而上考虑安全因素，其极简设计极大的减少了安全漏洞风险。

7 测试网络样例

Fabric-simple 仓库现在包含了一个新的测试网络。新测试网络采用模块化构建并提供用户友好的测试网络样例使得测试应用程序和链码变得更加容易。新测试网络除了 cryptogen 以外，也支持使用 CA 来部署网络。

有关新的测试网络的更多详情，请参阅使用 Fabric 测试网络一节。

8 升级到 Fabric 2.0

一个新的主要发布版本总会带来一些额外的升级担忧。不过请放心，Fabric v2.0 支持从 v1.4.x 进行滚动升级，因此，网络组件能够被升级而无需停止服务。升级文档做了相当多的扩充和重写，现在有了一个单独的主页入口 - 升级到最新版本。在这个主页中，可以找到诸如升级组件以及更新一个通道的能力等级这样的文档，也可以查阅 Fabric 2.0 升级须知。

发布说明为用户提供了更多有关 Fabric 2.0 的信息。特别是要看看那些变更以及宣布不再支持的特性和事项。更多详情，请参阅 Fabric 2.0 发布说明。
