本文汇总了一些针对 Fabric 性能基准测试的结论，有来自官方及第三方的测试数据和结论。其中包含使用 Hyperledger Caliper (一个来自官方的区块链性能基准测试工具) 对最新发布版本 (Fabric 1.4.4) 进行的一次性能基准实测的方法和结论。

1 Fabric 性能调查

1.1 来自 IBM 的论文

Fabric Paper - 来自 IBM 研究人员发布的论文 (发表于 2018 年 1 月) 性能报告：

Fabric Version Peer CPU Peer RAM 区块大小 合约 吞吐量 (TPS/s) 延迟 (ms)
v1.1 16c 8G 500 Fabcoin 3000 600
  • Fabcoin 合约实现类似于比特币功能的合约
  • 以上吞吐量和延迟均为挖矿交易测试
  • 测试显示 Orderer 共识网络的性能不是整体性能的瓶颈

1.2 来自官方的测试

来自 Hyperledger 官方使用 Caliper 对一个简单 Solo 网络执行测试的结果：

Fabric Version Peer CPU Peer RAM 区块大小 合约 背书策略 操作 吞吐量 (TPS/s) 延迟 (ms)
v1.4.0 16c (共享) 32G (共享) 500 Empty 1-of-N submit 485.4 100
v1.4.0 16c (共享) 32G (共享) 500 Empty 2-of-N query 792.3 40
  • 以上测试运行于一个 Solo 环境 (1 Orderer 节点；两个组织个一个 Peer 节点)
  • Empty 合约模拟最小化交易消息，交易内容为空，交易消息的大小为签名的大小
  • 背书策略 1-of-N 是指任意节点同意即可；2-of-N 是指两个组织均同意 (共 2 个节点)

详细的报告，请参阅 Hyperledger Blockchain Performance 文档。

1.3 来自第三方的性能测试和优化

以下是来自于第三方针对 Fabric V1.0 的测试及性能优化结果。

在经过三项优化后，测试结论如下：

  • LevelDB 无论吞吐量还是延迟都比 CouchDB 表现好；
  • 优化后吞吐量可达 2500 TPS/s
  • 优化后延迟 <= 40 ms
  • Block Size 对延迟的影响更显著

详细的报告，请参阅 Performance Benchmarking and Optimizing Hyperledger Fabric Blockchain Platform 一文。

2 测试方案

2.1 影响性能的因素

影响因素 因素趋势 性能趋势 说明
Orderer 共识网络 TPS + + 实测的结果是整个网络的性能瓶颈不在 Orderer 共识网络上
Peer 节点 CPU 资源 + + 系统吞吐量对 Peer 节点的 CPU 资源敏感
Peer 节点 Memory 资源 + + 重要但没有 CPU 资源那么敏感
区块批量大小 + + 快批量大有利于系统吞吐量，但会增加延迟
交易消息大小 + - 对吞吐量和延迟均有影响，尤其是存在 IO 性能瓶颈情况下
背书策略 + - 要求背书的节点越多，吞吐量越小，延迟越大
网络 IO + + 网络 IO 越高越好
磁盘 IO + + 磁盘 IO 越高越好，建议采用 SSD 磁盘

2.2 测试基准

为了获得实用的性能参照，我们搭建一个可用于生产环境的网络 (2 个组织、4 个 Peer 节点、5 个 Orderer 节点)，整个测试网络拓扑图及测试基准条件如下所示。

  • Caliper 版本：v0.20.0
  • Caliper 节点配置
    • CPU: 16c (启动 16 个 Client App)
    • RAM: 16G
    • Network: 3G link
  • Fabric 版本：v1.4.4
  • Orderer 共识网络 - 5 Etcd 节点 (Raft 算法)
  • 两个组织 org1 和 org2，每个组织各两个 Peer 节点
  • Peer 节点配置
    • CPU: 16c (性能对 Peer 节点 CPU 敏感)
    • RAM: 16G;
    • Network: 3G link
  • 状态数据库
    • LevelDB
  • 背书策略
    • 1-of-N (任意组织任意一个节点)
    • 2-of-N (两组织各一个节点)
  • 智能合约
    • High Throughput 合约 -
    • Empty 合约
  • 块大小
    • 500 交易
  • 交易消息大小
    • Empty 合约
      • Submit 交易消息大小: 4.5 kB
      • Query 交易消息大小： 2.6 kB
    • High Throughput 合约
      • Submit 交易消息大小: 9.1 kB
      • Query 交易消息大小： 2.7 kB
  • 网络 IO 环境 - 采用典型万兆数据中心网络
  • 磁盘 IO - 建议采用高速 SSD 硬盘

2.5 High Throughput 合约测试

2.5.1 High Throughput 合约

High Throughput 是 Fabric Samples 项目提供的高吞吐量智能合约的例子。我们采用这个智能合约进行性能测试。有关该合约的详情，请参阅High-Throughput Network。

2.5.2 测试脚本

如下分别是用于更新和查询的测试脚本。

  1. 用于更新的测试脚本
```js
'use strict';

module.exports.info  = 'opening accounts';

let account_array = [];
let txnPerBatch;
let initOperation;
let bc, contx;
module.exports.init = function(blockchain, context, args) {
    if(!args.hasOwnProperty('operation')) {
        return Promise.reject(new Error('htcc.operation - \'money\' is missed in the arguments'));
    }

    if(!args.hasOwnProperty('txnPerBatch')) {
        args.txnPerBatch = 1;
    }
    initOperation  = args.operation;
    txnPerBatch = args.txnPerBatch;
    bc = blockchain;
    contx = context;

    return Promise.resolve();
};

const dic = 'abcdefghijklmnopqrstuvwxyz';
/**
 * Generate string by picking characters from dic variable
 * @param {*} number character to select
 * @returns {String} string generated based on @param number
 */
function get26Num(number){
    let result = '';
    while(number > 0) {
        result += dic.charAt(number % 26);
        number = parseInt(number/26);
    }
    return result;
}

let prefix;
/**
 * Generate unique account key for the transaction
 * @returns {String} account key
 */
function generateAccount() {
    // should be [a-z]{1,9}
    if(typeof prefix === 'undefined') {
        prefix = get26Num(process.pid);
    }
    return prefix + get26Num(account_array.length+1);
}

function random() {
    return Math.random();
}

/**
 * Generates simple workload
 * @returns {Object} array of json objects
 */
function generateWorkload() {
    let workload = [];
    for(let i= 0; i < txnPerBatch; i++) {
        let acc_id = generateAccount();
        account_array.push(acc_id);

        if (bc.bcType === 'fabric') {
            workload.push({
                chaincodeFunction: 'update',
                chaincodeArguments: [acc_id, random().toString(),initOperation.toString()],
            });
        } else {
            workload.push({
                'verb': 'update',
                'account': acc_id,
                'operation': initOperation
            });
        }
    }
    return workload;
}

module.exports.run = function() {
    let args = generateWorkload();
    return bc.invokeSmartContract(contx, 'htcc', '1.0', args, 30);
};

module.exports.end = function() {
    return Promise.resolve();
};

module.exports.account_array = account_array;
```
  1. 用于查询的测试脚本
```js
'use strict';

module.exports.info  = 'querying accounts';

let bc, contx;
let account_array;

module.exports.init = function(blockchain, context, args) {
    const open = require('./update.js');
    bc       = blockchain;
    contx    = context;
    account_array = open.account_array;

    return Promise.resolve();
};

module.exports.run = function() {
    const acc  = account_array[Math.floor(Math.random()*(account_array.length))];

    if (bc.bcType === 'fabric') {
        let args = {
            chaincodeFunction: 'get',
            chaincodeArguments: [acc],
        };

        return bc.bcObj.querySmartContract(contx, 'htcc', '1.0', args, 10);
    } else {
        // NOTE: the query API is not consistent with the invoke API
        return bc.queryState(contx, 'htcc', '1.0', acc);
    }
};

module.exports.end = function() {
    // do nothing
    return Promise.resolve();
};
```

3 测试结果

  • 以上测试使用 Fabric v1.4.4 版本（最新版本）
  • 采用 Caliper 0.2.0 版本

更多测试条件，请参考测试基准一节。

  • 相同测试条件下 (Solo 环境) ，我们的测试中，查询交易性能优于官方数据；
  • 相同测试条件下 (Solo 环境)，我们的测试中，提交交易性能吞吐量和官方测试相仿，但延迟远大于官方数据，可能是磁盘 IO 对延迟影响较大；
  • 更高资源的 Peer 节点会增加吞吐量，特别是查询交易的吞吐量；
